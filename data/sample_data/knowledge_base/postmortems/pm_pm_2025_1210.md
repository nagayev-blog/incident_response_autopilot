# Postmortem: PM-2025-1210

**Title:** Kafka Consumer Lag 1.2M — SMS Provider Partial Outage Cascade  
**Date:** 2025-12-10  
**Severity:** CRITICAL  
**Duration:** 4 hours 30 minutes  
**Service:** notification-service  
**Author:** p.ivanova@company.ru  

---

## Executive summary

Third-party SMS provider experienced a partial outage, increasing SMS delivery latency from 200ms to 12 seconds. `notification-service` had no circuit breaker and no thread pool isolation between notification channels. Slow SMS calls blocked Kafka consumer threads, causing consumer lag to grow to 1.2M messages. 180,000 notifications (SMS, email, push) were delayed by 6+ hours for 130,000 users.

## Root cause

### Thread pool exhaustion cascade

`notification-service` used a single shared thread pool for all notification channels:

```
Kafka Consumer Thread Pool (3 threads × 3 partitions = 9 consumer threads)
         ↓
NotificationProcessor (same threads)
         ↓
SMSService.send()  ← BLOCKING CALL, 12s per SMS during outage
EmailService.send()  ← ALSO BLOCKED (same thread pool)
PushService.send()   ← ALSO BLOCKED (same thread pool)
```

When SMS latency jumped to 12s:
- Each consumer thread spent 12s blocked on SMS calls
- Kafka consumer poll interval exceeded (max.poll.interval.ms = 5000ms)
- Consumer group triggered partition rebalance
- Rebalance caused more delays, making lag worse
- Eventually all 9 consumer threads blocked on slow SMS calls
- Email and Push notifications (which were fast) also stopped

### No circuit breaker

Zero protection against external service degradation. No timeout shorter than SMS provider's own timeout (~30s). Connection hung until provider's TCP keepalive timeout.

---

## Fix

### Immediate (hotfix deployed during incident)
```yaml
# Reduce SMS call timeout from 30s to 3s
sms.provider.timeout.ms: 3000
# Fast-fail SMS calls, add to retry queue instead of blocking
```

### Permanent (deployed next sprint)

1. **Separate thread pools per channel:**
```java
@Bean("smsExecutor")
public Executor smsExecutor() {
    ThreadPoolTaskExecutor e = new ThreadPoolTaskExecutor();
    e.setCorePoolSize(5);
    e.setMaxPoolSize(10);
    e.setQueueCapacity(1000);
    e.setThreadNamePrefix("sms-");
    return e;
}

// Email and Push get their own pools — SMS slowdown cannot affect them
```

2. **Circuit breaker on SMS service:**
```java
@CircuitBreaker(name = "smsProvider",
    fallbackMethod = "smsFallback")
@TimeLimiter(name = "smsProvider")
public CompletableFuture<SmsResult> sendSms(SmsRequest req) {
    return CompletableFuture.supplyAsync(
        () -> smsClient.send(req), smsExecutor);
}

// Fallback: add to retry queue with 15-minute delay
public CompletableFuture<SmsResult> smsFallback(SmsRequest req, Throwable t) {
    retryQueue.addWithDelay(req, Duration.ofMinutes(15));
    return CompletableFuture.completedFuture(SmsResult.queued());
}
```

3. **Permanent consumer scaling:** 3 → 6 instances (headroom for external service degradation)

4. **Kafka consumer config tuning:**
```properties
max.poll.interval.ms=30000   # was 5000 — too aggressive for notification workloads
max.poll.records=100         # was 500 — reduce batch size to limit per-poll time
```

## What went wrong

1. No thread pool isolation between notification channels
2. No circuit breaker on any external service calls
3. Kafka `max.poll.interval.ms` too aggressive for I/O-bound workload
4. SMS provider degradation alert was in monitoring but not paged to on-call
5. No lag alert below 50k — could have been caught at 5-10k lag (1-2 minutes in)

## Lessons learned

1. **Thread pool isolation is non-negotiable for multi-channel services.** One slow channel must not block all channels.
2. **Every external I/O call needs a circuit breaker.** Not optional for production systems.
3. **Kafka consumer health depends on poll interval matching actual processing time.** Tune conservatively.
4. **External provider incidents should page on-call.** We had the data, we just didn't act on it fast enough.
