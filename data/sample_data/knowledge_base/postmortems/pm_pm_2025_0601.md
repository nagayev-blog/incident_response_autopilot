**Date:** 2025-06-01  
**Severity:** HIGH

**Service:** Keycloak 21.x (IAM), all authenticated services impacted  
**Author:** m.sorokina@company.ru

## Executive summary

Keycloak's built-in session cleanup task stopped running silently after the v20→v21 upgrade 6 weeks prior. The USER_SESSION table grew to 38M rows (35M expired). Sequential scans on this table during authentication caused login latency to reach 12 seconds at peak, with 12% of requests returning 504. Fixed by batch deletion of expired sessions over ~2 hours.

## Root cause

Keycloak 21.0 refactored its session management SPI and renamed the configuration key controlling session cleanup scheduling. The old key (`spi-user-session-infinispan-*`) was silently ignored after upgrade. The new key was not in our deployment config.

Session cleanup had been running every night in v20. In v21, it simply stopped — no errors, no alerts, no log output. Over 47 days, sessions accumulated at ~800k/day.

The USER_SESSION table had no index on `last_session_refresh`. Every authentication request performs:
```sql
SELECT * FROM USER_SESSION WHERE realm_id = ? AND user_id = ?
-- With 38M rows, this becomes a sequential scan: ~4-8 seconds
```

---

## Fix applied

1. **Immediate:** Batch DELETE of expired sessions (sessions older than 24h):
   - 35M rows deleted in ~110 minutes (500k per batch to avoid lock escalation)
   - Table reduced from 38M → 3.2M rows

2. **Index added** (ran CONCURRENTLY to avoid locking):
   ```sql
   CREATE INDEX CONCURRENTLY idx_user_session_refresh 
   ON USER_SESSION(last_session_refresh);
   ```

3. **Keycloak config updated** with correct v21 session cleanup keys:
   ```properties
   spi-session-infinispan-remote-session-cache-entries-expiry=true
   ```

4. **Realm settings tightened:**
   - SSO Session Max: 24h → 8h
   - Offline Session Max: 180d → 30d
   - Revoke Refresh Token: enabled

5. **Weekly maintenance CronJob** added:
   ```yaml
   schedule: "0 3 * * 0"  # Every Sunday 03:00
   ```

## What went well

1. DBA identified sequential scan root cause quickly (30 min from alert)
2. Batch deletion approach avoided making the situation worse (no lock escalation)
3. Index creation CONCURRENTLY — no additional downtime

## What went wrong

1. Keycloak upgrade runbook had no step to verify session cleanup is working post-upgrade
2. No alert on USER_SESSION table size
3. Session cleanup failure was completely silent — no logs, no metrics
4. 47 days to detect — would have become CRITICAL soon regardless

## Lessons learned

1. **Silent failures in maintenance tasks are dangerous.** Log successful execution. Alert on absence of execution.
2. **Upstream upgrades need specific functional verification.** Not just "is the service running?" but "are all background tasks running?"
3. **Database table growth should be monitored.** Tables > 10M rows without expected growth rate should alert.
