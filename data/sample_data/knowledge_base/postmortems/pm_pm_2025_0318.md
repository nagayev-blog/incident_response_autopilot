## Executive summary

Payment service v2.9.0 introduced a connection leak in the new payment gateway retry handler. Under normal load the leak was invisible (connections returned slowly). During the morning peak, the PostgreSQL connection pool exhausted within 28 minutes of deployment, causing 87% HTTP 503 error rate on all payment endpoints. Revenue impact ~12.3M RUB in 47 minutes. Resolved by rollback to v2.8.7.

## Root cause

New retry logic in `PaymentGatewayService.java` acquired a DB connection at the start of the `processPayment()` method. When the external payment gateway returned HTTP 429 (rate limit), the exception handler added the request to the retry queue and returned early — but did not close the DB connection.

```java
// BUGGY CODE — v2.9.0
public PaymentResult processPayment(PaymentRequest req) {
    Connection conn = dataSource.getConnection();  // ← acquired here
    try {
        PaymentResponse resp = gatewayClient.charge(req);

        if (resp.getStatus() == 429) {
            retryQueue.add(req);           // ← returns early
            return PaymentResult.queued(); // ← connection LEAKED (never closed)
        }

        return buildResult(conn, resp);
    } catch (Exception e) {
        log.error("Payment failed", e);
        throw e;                           // ← also leaks on generic exceptions
    }
    // No finally block — connection not returned to pool on any exception path
}
```

Under normal load: leak rate ~1 connection/minute, went unnoticed for 28 minutes.  
At peak load: leak rate ~5 connections/minute, pool exhausted in 11 minutes.

---

## Fix

```java
// FIXED — v2.9.1 (deployed same day at 11:30)
public PaymentResult processPayment(PaymentRequest req) {
    // Option 1: Use @Transactional (Spring manages connection lifecycle)
    // Option 2: try-with-resources explicit close

    try (Connection conn = dataSource.getConnection()) {
        PaymentResponse resp = gatewayClient.charge(req);

        if (resp.getStatus() == 429) {
            retryQueue.add(req);
            return PaymentResult.queued();
            // conn.close() called automatically by try-with-resources
        }

        return buildResult(conn, resp);
    }
    // Connection always returned to pool — no leak possible
}
```

## What went well

1. Alerting fired within 8 minutes of deploy
2. `kubectl rollout history` immediately showed the correlation with deploy time
3. Rollback procedure was practiced and executed correctly
4. On-call responded within 3 minutes of page

## What went wrong

1. No connection pool saturation test in staging pipeline
2. Code review did not catch the missing `finally` block
3. Alert threshold (80 connections) set too high — should alert at 70% pool usage
4. No `hikaricp_connections_pending` alert — early warning would have helped

## Lessons learned

1. **Retry logic is a common source of resource leaks.** Any new retry/fallback code path must be reviewed specifically for resource cleanup.
2. **Static analysis catches this class of bug at zero cost.** SpotBugs rule should have been in pipeline from day one.
3. **Deploy + alert correlation in Grafana is critical.** Deploy annotations on Grafana dashboards would have cut MTTR by 15 minutes.
