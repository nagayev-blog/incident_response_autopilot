# Postmortem: PM-2025-0915

**Title:** ETL Pipeline Deadlock During Monthly Risk Score Batch  
**Date:** 2025-09-15  
**Severity:** HIGH  
**Duration:** 5 hours data freshness lag; ETL itself ran 4h past schedule  
**Service:** data-sync-service (ETL), risk-score-batch  
**Author:** n.kuznetsova@company.ru  

---

## Executive summary

Monthly risk score recalculation batch job and nightly ETL sync both lock the same PostgreSQL tables (`client_profiles`, `transaction_history`). No scheduling coordination existed. ETL retried 3 times then gave up after each attempt deadlocked within minutes. Analytics data was 5 hours stale for the morning business reporting window.

## Root cause

Two independently scheduled jobs with table-level lock requirements:

```
risk-score-batch:
  Schedule: 1st of month, 01:00
  Duration: ~3-4 hours  
  Locks: client_profiles (exclusive), transaction_history (exclusive)

etl-sync:
  Schedule: daily 01:30
  Duration: ~30 minutes normally
  Locks: client_profiles (shared read), transaction_history (shared read)
```

PostgreSQL deadlock scenario:
1. `risk-score-batch` holds EXCLUSIVE lock on `transaction_history`, waits for `client_profiles`
2. `etl-sync` holds SHARE lock on `client_profiles`, waits for `transaction_history`
3. PostgreSQL detects deadlock, kills the lighter transaction (ETL)

ETL retry logic had 5-minute backoff and 3 attempts — all three attempts landed within the batch window.

---

## Fix

1. **ETL rescheduled to 06:00** — safe window after any monthly batch completes
2. **Batch job pre-check added** — ETL checks for running batch before acquiring locks:
   ```python
   def is_batch_running() -> bool:
       result = db.execute("""
           SELECT count(*) FROM pg_stat_activity 
           WHERE application_name = 'risk-score-batch' AND state != 'idle'
       """)
       return result.scalar() > 0
   
   if is_batch_running():
       logger.warning("Batch job running, delaying ETL by 30 minutes")
       time.sleep(1800)
   ```
3. **Exponential backoff improved:** 3 retries at 5min → 5 retries at 10/20/40/80/120min
4. **Alert added:** `etl_last_success_lag > 3h`
5. **Grafana annotation added:** batch job start/end shown on analytics dashboards

## Lessons learned

1. **Batch jobs and streaming ETL must coordinate on lock dependencies.** Add lock dependency map to runbook.
2. **Retry intervals must outlast the expected lock hold time.** 5 min backoff useless against 4h batch job.
3. **Silent ETL failures are high risk.** Email/Slack notification on ETL failure, not just Prometheus alert.
