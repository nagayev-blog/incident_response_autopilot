# Postmortem: PM-2024-1102

**Title:** Report Generation Service OOMKilled — Month-End Reporting Outage  
**Date:** 2024-11-02  
**Severity:** HIGH  
**Duration:** 2 hours 15 minutes  
**Service:** report-generation-service v1.8.3  
**Author:** a.fedorov@company.ru  

---

## Executive summary

New "full portfolio export" feature (v1.8.3) loaded entire multi-year transaction datasets into a `List<Map>` before streaming to XLSX. Reports with >500k rows caused JVM heap to exceed the 512Mi container limit. Pod entered CrashLoopBackOff, all report generation failed for 2h15m during month-end reporting peak. Fixed by streaming approach and memory limit increase.

## Root cause

```java
// report-generation-service/ReportExportHandler.java — v1.8.3 BUGGY
public ResponseEntity<byte[]> exportPortfolio(ExportRequest req) {
    // PROBLEM: queryForList loads ALL rows into memory at once
    List<Map<String, Object>> rows = jdbcTemplate.queryForList(
        "SELECT * FROM transactions WHERE client_id = ? AND date >= ?",
        req.getClientId(), req.getFromDate()
    );
    // For 2.4M rows × ~200 bytes/row = 480MB just for the List
    // Plus XLSX builder overhead = OOM at 512Mi limit

    byte[] xlsx = xlsxBuilder.build(rows);  // another copy in memory
    return ResponseEntity.ok(xlsx);
}
```

Memory consumption per request:
- 2.4M rows × ~200 bytes average = ~480 MB for `List<Map>`
- XLSX builder working memory: additional ~150 MB
- Total: ~630 MB → exceeds 512 Mi limit

---

## Fix

```java
// v1.8.4 — Streaming approach, constant memory usage regardless of dataset size
public StreamingResponseBody exportPortfolio(ExportRequest req) {
    return outputStream -> {
        SXSSFWorkbook wb = new SXSSFWorkbook(100); // keep 100 rows in memory
        Sheet sheet = wb.createSheet("Transactions");
        AtomicInteger rowNum = new AtomicInteger(0);

        jdbcTemplate.query(
            "SELECT * FROM transactions WHERE client_id = ? AND date >= ?",
            rs -> {
                Row row = sheet.createRow(rowNum.getAndIncrement());
                // write row directly to streaming workbook
                writeRow(row, rs);
                if (rowNum.get() % 10000 == 0) {
                    ((SXSSFSheet) sheet).flushRows(10000); // flush to disk
                }
            },
            req.getClientId(), req.getFromDate()
        );

        wb.write(outputStream);
        wb.dispose();  // cleanup temp files
    };
}
```

Additionally added: row count guard (HTTP 413 if > 2M rows requested, with pagination hint).

## Lessons learned

1. **Never load unbounded datasets into memory.** All bulk operations must use streaming or pagination.
2. **Month-end is peak load for reports.** New export features need load testing before month-end deploys.
3. **Feature flags for new endpoints.** Would have allowed disabling export without full rollback.
