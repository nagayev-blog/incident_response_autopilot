То есть ResponseAgent получает на вход три потока данных одновременно:

Источник	Что даёт
DiagnosisAgent	Что сломалось и почему
HistoryAgent (RAG)	Как похожее решали раньше, какой runbook подходит
engineer_feedback	Замечания оператора (если был Reject)
На скрине видно, что History нашёл rb_kafka_consumer_lag.md со score 0.86 — и ResponseAgent, получив этот runbook в контексте, сформирует план именно по шагам из него, а не из головы.

Практический эффект: без RAG план был бы общим ("проверьте логи, увеличьте heap"). С RAG — конкретным ("выполните шаги из runbook RB-KAFKA-001: проверьте max.poll.interval.ms, сделайте rolling restart consumer group").

### увидел в инструменте пока только трассировку состояний (state)
СС назвал причину: LangSmith автоматически трейсит LangGraph-узлы (input/output state), но не видит внутренности — то есть сами вызовы client.messages.parse() с промптами.

Причина: агенты используют Anthropic SDK напрямую, не через LangChain. LangSmith не инструментирует его автоматически.

Чтобы промпты были видны, нужно обернуть LLM-вызовы декоратором @traceable. Пример для одного агента:

from langsmith import traceable

@traceable(name="ResponseAgent LLM call")
def _call_llm(system: str, user: str) -> ResponseOutput:
    response = _client.messages.parse(
        model=settings.triage_model,
        system=system,
        messages=[{"role": "user", "content": user}],
        ...
    )
    return response.parsed_output
Тогда в LangSmith появится отдельный span внутри узла с полным промптом.
